Redis 的字典采用哈希表作为底层实现， 一个哈希表里面可以有多个哈希表节点， 而每个哈希表节点就保存了字典中的一个键值对。依次来分析一下哈希表、哈希节点、以及字典的结构。

结构

字典 ：type和privdata是为了多态的创建不同的字典，dictht是一个双指针rehash的时候会使用到

哈希表：table 是一个数组，元素是一个指针指向dictEntry

哈希节点：dictEntry保存着一个键值对

遍历

安全的迭代器在遍历过程中允许删除元素，意味着字典第一维数组下面挂接的链表中的元素可能会被摘走，元素的 next 指针就会发生变动，这是否会影响迭代过程呢？下面我们仔细研究一下迭代函数的代码逻辑。

大部分逻辑都已经注释上了，整个方法是一个死循环，如果 entry 等于 null，要么是迭代器初次工作，要么是迭代到一个桶的最后节点处了。如果是后者，会进入 if 逻辑中，判断是否整个字典全部迭代结束，如果不是取下一个桶。

如果字典未处于 rehash 状态，自增 iterators 属性的操作会禁止后续节点操作触发 rehash，如果已经处于 rehash 过程了，也不慌，当前 ht[0] 迭代结束后，再去迭代早于迭代器工作前已经被转移到 ht[1] 的那些节点。因为如果你是安全迭代器的话，iterators 一自增之后，后续节点就不会触发 rehash 迁移节点，所以不会重复迭代数据。

值得注意的是在字典扩容时进行rehash，将旧数组中的链表迁移到新的数组中。某个具体槽位下的链表只可能会迁移到新数组的两个槽位中。

解决冲突

如果两个不同的可以被哈希到同一个桶中，则会通过dictEntry的next链起来形成一个链表，这里为了性能考虑是直接链到链表的头部。

重哈希

上面提到解决哈希冲突的方式是链地址法，当这个哈希表接近满的时候会有大量的key被哈希到同一个桶中，形成一个长链表遍历的时候时间复杂度很高，这个时候需要重新洗牌，申请一个更大的table重新计算哈希值。如果数据量很大时，由于redis单进程机制可能会导致段时间停止服务。redis采取的措施是新建一个哈希表作为缓冲，双读单写，新数据来时插入新表有访问请求时先读旧表，读不到就去新表读，全部迁移完后就把旧表删除。

ht[1]就是那个缓冲新表，如果是扩容ht[1]需要分配的大小为第一个不大于等于ht[0].used*2的2^n, 如果是缩容ht[1]的大小为第一个不大于等于ht[0].usedd的2^n

扩容和所容是装载因子触发的，当装载因子大于等于1时会触发扩容，当转载因子小于0.1时会触发缩容。

装载因子的计算：load_factor = ht[0].used / ht[0].size

另外当服务器正在执行BGSAVE或BGREWRITEAOF命令时装载因子会被提高，因为执行这些命令时会fork只进程去处理大多数操作系统采用写时复制，为了避免不必要的内存写入操作，尽可能的将扩容往后推。

上面提到的rehash操作不是一次性，集中完成，而是分批，渐进式的完成的，rehash的好处是采取分而治之的方式，将迁移工作分摊到增删改查操作上。
